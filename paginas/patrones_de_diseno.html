Patrones de diseño

¿Qué son los patrones de diseño?

Los patrones de diseño son soluciones habituales a problemas que ocurren con frecuencia en el diseño de software. Son como planos prefabricados que se pueden personalizar para resolver un problema de diseño recurrente en el código.

No se puede elegir un patrón y copiarlo en el programa como si se tratara de funciones o bibliotecas ya preparadas. El patrón no es una porción específica de código, sino un concepto general para resolver un problema particular. Se pueden seguir los detalles del patrón e implementar una solución que encaje con las realidades del propio programa.

A menudo los patrones se confunden con algoritmos porque ambos conceptos describen soluciones típicas a problemas conocidos. Mientras que un algoritmo siempre define un grupo claro de acciones para lograr un objetivo, un patrón es una descripción de más alto nivel de una solución.

Los 3 patrones más comunes

1. Singleton (Creacional)

El patrón Singleton asegura que una clase tenga una única instancia y proporciona un punto de acceso global a dicha instancia. Es útil cuando se necesita controlar el acceso a algún recurso compartido, como una base de datos o un archivo de configuración.

Para implementarlo, se hace privado el constructor y se crea un método estático que devuelva siempre la misma instancia. Un ejemplo clásico es una conexión a base de datos que debe ser única en toda la aplicación.

  • Garantiza una única instancia de la clase
  • Proporciona acceso global al objeto
  • Se inicializa solo cuando se necesita por primera vez

2. Factory Method (Creacional)

El patrón Factory Method proporciona una interfaz para crear objetos en una superclase, mientras permite a las subclases alterar el tipo de objetos que se crearán. En lugar de llamar al operador new directamente, se invoca a un método fábrica especial.

Un ejemplo práctico es un sistema de logística: la clase base define el método fábrica para crear transportes, y las subclases concretas deciden si crear un camión o un barco, devolviendo objetos que implementan la misma interfaz.

  • Evita acoplamiento fuerte entre creador y productos
  • Facilita agregar nuevos tipos de productos
  • Sigue el principio de responsabilidad única

3. Observer (Comportamiento)

El patrón Observer define un mecanismo de suscripción para notificar a varios objetos sobre cualquier evento que le suceda al objeto que están observando. El objeto observado se llama notificador y los que reciben las notificaciones son los suscriptores.

Un ejemplo cotidiano es una suscripción a un periódico: el notificador mantiene una lista de suscriptores y les envía nuevos números cuando se publican. Los suscriptores pueden abandonar la lista en cualquier momento.

  • Permite notificar a múltiples objetos sobre cambios
  • Los suscriptores pueden unirse o abandonar dinámicamente
  • Desacopla el notificador de los suscriptores

Ejemplo: Patrón Observer

En este ejemplo, un editor de texto notifica a otros objetos tipo servicio sobre los cambios en su estado. La lista de suscriptores se compila dinámicamente: los objetos pueden empezar o parar de escuchar notificaciones durante el tiempo de ejecución. 


class EventManager {
    private listeners: hash map

    subscribe(eventType, listener) {
        listeners.add(eventType, listener)
    }

    unsubscribe(eventType, listener) {
        listeners.remove(eventType, listener)
    }

    notify(eventType, data) {
        foreach (listener in listeners.of(eventType)) {
            listener.update(data)
        }
    }
}

class Editor {
    public events: EventManager
    private file: File

    constructor() {
        events = new EventManager()
    }

    openFile(path) {
        this.file = new File(path)
        events.notify("open", file.name)
    }

    saveFile() {
        file.write()
        events.notify("save", file.name)
    }
}

interface EventListener {
    update(filename)
}

class LoggingListener implements EventListener {
    private log: File
    private message: string

    constructor(log_filename, message) {
        this.log = new File(log_filename)
        this.message = message
    }

    update(filename) {
        log.write(replace('%s', filename, message))
    }
}

class EmailAlertsListener implements EventListener {
    private email: string
    private message: string

    constructor(email, message) {
        this.email = email
        this.message = message
    }

    update(filename) {
        system.email(email, replace('%s', filename, message))
    }
}

class Application {
    config() {
        editor = new Editor()

        logger = new LoggingListener(
            "/path/to/log.txt",
            "Someone has opened the file: %s")
        editor.events.subscribe("open", logger)

        emailAlerts = new EmailAlertsListener(
            "admin@example.com",
            "Someone has changed the file: %s")
        editor.events.subscribe("save", emailAlerts)
    }
}
